# NeverBroke 微信小程序详细需求文档

> **产品名称**：NeverBroke（认知翻身账）  
> **平台**：微信小程序  
> **版本**：v1.0  
> **文档日期**：2026年3月29日

---

## 一、产品概述

### 1.1 产品定位
NeverBroke 是一款面向个人用户的现代化记账与财务规划微信小程序，帮助用户清晰管理收支、预算、分期付款，并对未来现金流和税后收入形成可操作的预期。

### 1.2 产品愿景
让用户在一个小程序内完成：**记账 → 预算与分期 → 周期收入与工资税后测算 → 未来现金流与时间线视图**，减少在多个 App 和表格间切换的成本。

### 1.3 核心价值
- **可信的个人账本**：多账户、多币种、信用卡额度管理
- **可计划的支出**：月度预算、分期管理
- **可预期的现金**：周期收入投射、现金流预测
- **可理解的复盘**：月度报表、分类统计、趋势分析
- **安全与隔离**：用户数据完全隔离，支持多设备同步

## 二、目标用户与使用场景

### 2.1 目标用户画像

#### 用户群体 1：工薪族
- **特征**：有固定工资收入，日常消费较规律
- **痛点**：不清楚钱花在哪里，月底总是紧张
- **需求**：记日常账、设月度预算、查看本月报表

#### 用户群体 2：有分期需求的用户
- **特征**：使用信用卡分期、花呗分期等
- **痛点**：多个分期计划难以管理，不知道每月要还多少
- **需求**：管理信用卡分期、预览大额分期、查看还款节奏

#### 用户群体 3：关心税后收入的用户
- **特征**：想了解五险一金扣除后的实际到手工资
- **痛点**：不清楚社保公积金扣多少，税后到手不明确
- **需求**：配置城市社保规则、工资方案，查看税后预览

#### 用户群体 4：有大额支出规划的用户
- **特征**：近期有买房、买车、旅游等大额支出计划
- **痛点**：不确定未来几周资金是否充足
- **需求**：使用时间线与现金流预测，判断资金安全性

### 2.2 典型使用场景

**场景 1：日常记账**
> 小王在便利店买了早餐，打开小程序，点击"记一笔"，选择"支出"，输入金额 15 元，选择分类"餐饮"，选择支付账户"微信零钱"，保存。整个过程不超过 10 秒。

**场景 2：查看本月预算使用情况**
> 月中时，小王想知道本月预算还剩多少。打开小程序首页，即可看到"本月预算"卡片，显示已用 2500/5000 元，进度条显示 50%，还有 2500 元可用。

**场景 3：创建分期计划**
> 小王买了一台 6000 元的笔记本电脑，想分 6 期还款。在记账时选择"创建分期"，输入总金额 6000 元，选择 6 期，系统自动计算每期 1000 元，并生成未来 6 个月的还款计划。

**场景 4：查看未来现金流**
> 小王计划下月旅游，想知道账户余额是否充足。打开"现金流预测"页面，查看未来 30/60/90 天的现金流曲线，系统提示"30 天后余额可能低于安全线，建议预留资金"。

**场景 5：工资税后计算**
> 小王想知道月薪 10000 元在北京的税后到手工资。打开"薪资控制"页面，选择城市"北京"，输入税前工资 10000 元，系统自动计算五险一金、个税，显示税后到手约 7800 元。

**交互流程**：

**流程 1：微信授权登录（推荐，默认方式）**
1. 用户打开小程序，自动进入登录页
2. 点击"微信一键登录"按钮
3. 调用 `wx.login()` 获取 code
4. 调用 `wx.getUserProfile()` 获取用户信息（昵称、头像）
5. 发送到后端 `/api/oauth/wechat/login`
6. 后端逻辑：
   - 如果微信 openid 已存在，直接登录
   - 如果是新用户，自动创建账号并登录
   - 返回 Access Token 和 Refresh Token
7. 小程序存储 Token 到本地
8. 跳转到首页

**流程 2：手机号登录（备用方式）**
1. 用户点击"其他登录方式"
2. 展开手机号登录表单
3. 输入手机号，点击"获取验证码"
4. 输入验证码，点击"登录"
5. 调用 `/api/auth/phone/login`
6. 后端验证验证码，返回 Token
7. 跳转到首页

**首次登录自动创建账号**：
- 微信登录：使用微信昵称和头像自动创建账号
- 手机号登录：使用手机号自动创建账号
- 无需单独注册流程

#### 5.1.3 API 接口

| 接口 | 方法 | 路径 | 说明 |
|-----|------|------|------|
| 微信登录 | POST | `/api/auth/wechat/login` | 微信一键登录（主要方式） |
| 手机号登录 | POST | `/api/auth/phone/login` | 手机号验证码登录（备用） |
| 发送验证码 | POST | `/api/auth/phone/send-code` | 发送手机验证码 |
| 刷新Token | POST | `/api/auth/refresh` | 刷新 Access Token |
| 登出 | POST | `/api/auth/logout` | 登出当前设备 |
| 全设备登出 | POST | `/api/auth/logout_all` | 登出所有设备 |

**微信登录接口详细说明**：

**请求示例**：
```json
POST /api/auth/wechat/login
{
  "code": "081xYz0w3...",
  "nickname": "张三",
  "avatar": "https://..."
}
```

**响应示例**：
```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "access_token": "eyJhbGc...",
    "refresh_token": "eyJhbGc...",
    "user": {
      "id": "123",
      "nickname": "张三",
      "avatar": "https://...",
      "is_new_user": false
    }
  }
}
```

**后端处理逻辑**：
1. 使用 code 向微信服务器换取 openid
2. 查询数据库是否存在该 openid
3. 如果存在：更新用户信息，返回 Token
4. 如果不存在：创建新用户，返回 Token 并标记 `is_new_user: true`
5. 新用户首次登录后，引导完成初始设置（创建第一个账户）

## 六、技术实现方案

### 6.1 小程序技术栈

#### 6.1.1 前端技术

- **UI组件库**：WeUI / Vant Weapp
- **状态管理**：小程序全局状态 / MobX（可选）

#### 6.1.2 后端技术
- **语言**：Go 1.20+
- **框架**：Gin
- **数据库**：PostgreSQL 14+
- **ORM**：GORM
- **认证**：JWT（Access Token + Refresh Token）
- **密码加密**：Argon2id
- **日志**：Uber Zap
- **构建工具**：Mage

### 6.2 数据库设计

#### 6.2.1 核心数据表（8张）

**1. users（用户表）**
```sql
- id: 主键
- phone: 手机号
- password_hash: 密码哈希
- nickname: 昵称
- avatar: 头像
- created_at, updated_at
```

**2. accounts（账户表）**
```sql
- id: 主键
- user_id: 用户ID（外键）
- name: 账户名称
- type: 账户类型（savings/credit/cash/investment）
- bank: 银行名称
- balance: 余额
- credit_limit: 信用卡额度
- currency: 币种
- archived: 软删除标记
- created_at, updated_at
```

**3. transactions（交易表）**
```sql
- id: 主键
- user_id: 用户ID（外键）
- account_id: 账户ID（外键）
- type: 交易类型（income/expense/transfer）
- amount: 金额
- category: 分类
- date: 交易日期
- note: 备注
- installment_plan_id: 分期计划ID（可选）
- created_at, updated_at
```

**4. installment_plans（分期计划表）**
```sql
- id: 主键
- user_id: 用户ID（外键）
- total_amount: 总金额
- installments: 分期数
- paid_installments: 已还期数
- start_date: 开始日期
- status: 状态
- created_at, updated_at
```

**5. recurring_incomes（周期收入表）**
```sql
- id: 主键
- user_id: 用户ID（外键）
- name: 名称
- amount: 金额
- frequency: 频率（weekly/monthly/yearly）
- next_due_date: 下次到期日期
- account_id: 账户ID（外键）
- created_at, updated_at
```

**6. budgets（预算表）**
```sql
- id: 主键
- user_id: 用户ID（外键）
- year: 年份
- month: 月份
- total_budget: 总预算
- created_at, updated_at
```

**7. user_settings（用户设置表）**
```sql
- id: 主键
- user_id: 用户ID（外键）
- language: 语言
- hide_amount: 是否隐藏金额
- default_currency: 默认货币
- notifications: 通知设置（JSON）
- created_at, updated_at
```

**8. refresh_tokens（刷新令牌表）**
```sql
- id: 主键
- user_id: 用户ID（外键）
- token_hash: Token哈希
- expires_at: 过期时间
- created_at
```

#### 6.2.2 索引设计
- `user_id` 字段建立索引（所有业务表）
- `user_id + date` 复合索引（transactions表）
- `user_id + status` 复合索引（installment_plans表）
- `user_id + next_due_date` 复合索引（recurring_incomes表）

```json

```

#### 6.3.2 响应格式
```json
{
```

- HTTPS 加密传输（生产环境必须）

#### 7.1.3 OAuth 安全

2. 考虑周期收入的固定时间点
3. 考虑分期还款的固定支出

#### 9.2.2 薪资计算准确性
**难点**：各地社保规则不同，个税计算复杂

**解决方案**：
1. 使用 social-data-tool 维护城市社保数据
2. 实现累计预扣法个税计算
3. 提供计算结果仅供参考的免责声明
4. 定期更新社保数据

#### 9.2.3 数据安全与隔离
**难点**：确保用户数据完全隔离

**解决方案**：
1. 所有业务表包含 user_id 字段
2. 所有查询必须带 user_id 条件
3. 使用 GORM 的 Scope 自动添加 user_id 过滤
4. 定期安全审计
